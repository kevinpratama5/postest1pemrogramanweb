🖥 Fullscreen Address Bar Spoofing PoC

CVE-2025-1019 · CVE-2024-8388 · CVE-2022-46877 · CVE-2023-25743 — Research Only
Tujuan: Sembunyikan atau bypass notifikasi fullscreen browser ("Press ESC to exit") lalu tampilkan fake Chrome UI dengan address bar yang menunjukkan accounts.google.com. User mengira sedang melihat browser asli.

Kenapa Notifikasi Harus Disembunyikan + Data Empiris

Saat requestFullscreen() dipanggil, browser menampilkan bar notifikasi di atas layar yang bilang "Press ESC to exit full screen". Notifikasi ini adalah sinyal utama bagi user bahwa mereka sedang dalam mode fullscreen — bukan melihat browser asli.

Jika notifikasi ini terlihat → user sadar → spoof gagal.
Jika notifikasi berhasil di-cover/disembunyikan atau user tidak sempat react → spoof berhasil.

⚠ EMPIRICAL TEST RESULTS (measured):
Chrome 148:  Notif visible 1175ms → browser-process layer → z-index cover TIDAK BISA
Firefox 149: Notif visible 1321ms → browser-process layer → z-index cover TIDAK BISA
Opera 131:   Notif visible 2376ms → browser-process layer → z-index cover TIDAK BISA

Arc:         [KRITIS] Tidak muncul notifikasi JIKA page sudah dalam fullscreen state!

REALITA:
Chrome/Edge/Opera/Firefox: Notifikasi dirender oleh browser process (compositor layer)
                           di ATAS semua web content. z-index berapapun tidak efektif.
Arc Browser:  Perilaku berbeda — saat requestFullscreen() dipanggil KEMBALI dari page
              yang sudah fullscreen, Arc tidak menampilkan notifikasi origin lagi.
              Exploit chain: auto-fullscreen saat load + F11 block + Teknik 1 = 0 notif!

STRATEGI YANG MASIH RELEVAN di 2025:
1. Arc Attack Chain (lihat card khusus di bawah) — 0 notifikasi, paling berbahaya
2. Form-First Attack (Teknik 7) — distract user selama 913-2376ms window
3. Social Engineering Normalization (Teknik 4) — fake ESC bar
4. F11 Block (Teknik 6) — berhasil di Firefox ✓ (confirmed)
5. Mobile browsers (Safari iOS, Chrome Android) — notif minimal

🟣 Arc Browser Attack Chain — KRITIS (0 Notifikasi)

Arc Browser memiliki perilaku unik: ketika halaman sudah dalam fullscreen dan requestFullscreen() dipanggil kembali, Arc tidak menampilkan notifikasi origin/URL. Berbeda dengan Chrome/Firefox/Edge/Opera yang selalu menampilkan notifikasi pada setiap fullscreen request.

ATTACK CHAIN:
  Step 1: Halaman dibuka (dari link click = navigation activation ada)
          → requestFullscreen() auto-dipanggil saat load → BERHASIL
          → Arc masuk fullscreen, page konten terlihat normal (PoC cards)
  Step 2: F11 diblock secara otomatis (auto-intercept aktif dari load)
          → User tidak bisa keluar via F11
  Step 3: Klik "Teknik 1 — Immediate Cover"
          → requestFullscreen() dipanggil LAGI (sudah fullscreen sebelumnya)
          → Di Arc: TIDAK ada notifikasi origin yang muncul!
          → Fake Chrome shell + accounts.google.com address bar tampil
  Step 4: User melihat fake Google login — mengira itu browser asli

KENAPA ARC BERBEDA:
  Arc menggunakan Chromium engine tapi dengan UI custom yang berbeda.
  Manajemen fullscreen notification di Arc tidak sama dengan Chrome upstream.
  "Re-fullscreen" dari state yang sudah fullscreen → Arc skip notifikasi.

IMPLIKASI:
  Di Chrome/Firefox/Edge/Opera: notifikasi selalu muncul 913-2376ms
  Di Arc: notifikasi tidak muncul sama sekali jika pre-fullscreen sudah ada
  → Severity: CRITICAL untuk Arc users

Teknik 1 — Immediate Cover (Paling Efektif)

Tampilkan fake toolbar dengan z-index maximum pada saat yang sama dengan requestFullscreen. Fake toolbar menutupi area di mana notifikasi akan muncul sebelum user sempat membacanya.

// Kunci: tampilkan cover SEBELUM fullscreen confirmed, bukan setelah
// Notifikasi Chrome muncul di ~top 0-60px layar
// Fake toolbar kita juga ada di top 0-52px dengan z-index max
// User melihat fake toolbar, notifikasi tertutup di belakangnya

Teknik 2 — Rapid Toggle Test (Event-Driven)

Enter fullscreen → langsung exit (80ms) → coba re-entry dari fullscreenchange handler. Test apakah browser mengizinkan requestFullscreen() tanpa fresh user gesture. Jika berhasil → notifikasi ke-2 mungkin tidak muncul.

// Kenapa loop async tidak bisa: requestFullscreen() butuh user gesture
// Setiap iterasi loop setelah yang pertama = tidak ada user gesture = Permissions check failed
//
// Fix: gunakan fullscreenchange event untuk chain:
// click (gesture) → enter fs → exit (80ms) → re-entry dari event handler
// → test apakah browser allow re-entry tanpa fresh gesture
//
// Histori: CVE-2022-46877 (Firefox ≤ 71) — notifikasi dirender sebagai web content
// → element dengan z-index max menutupi notifikasi. Dipatch di Firefox 72.

Teknik 3 — Element Fullscreen (bukan documentElement)

Request fullscreen pada elemen spesifik (div) bukan pada document.documentElement. Di beberapa browser, behavior notifikasi berbeda saat yang di-fullscreen adalah child element.

// Bedanya: element.requestFullscreen() vs documentElement.requestFullscreen()
// Di Firefox lama: element fullscreen menampilkan notifikasi lebih singkat
// Di Chrome: sama saja — tapi element fullscreen + pointer-events trick bisa beda
// Test di Arc Browser — sidebar UI mereka handle ini berbeda

Teknik 4 — Social Engineering + Fake ESC Bar

Tunjukkan dialog yang meyakinkan ("Sign in required") yang minta user klik. Setelah fullscreen, tampilkan fake ESC bar di bottom yang terlihat seperti notifikasi browser asli tapi sebenarnya dikendalikan oleh attacker.

// Fake ESC bar di bawah: "Press ESC to exit full screen"
// User melihatnya sebagai notifikasi browser biasa
// Tapi bar itu adalah HTML kita — kita bisa kontrol isinya, timing, dan tampilannya
// Teknik ini tidak menyembunyikan notifikasi asli, tapi MENORMALISASI presence-nya
// User tidak curiga karena "oh ada tulisan ESC, itu normal"

Teknik 5 — Pre-positioned Blocker (Notif Blocker)

Sebelum requestFullscreen, tempatkan sebuah div transparan dengan z-index maximum di posisi tepat di mana notifikasi akan muncul (top 0). Div ini sudah ada duluan saat notifikasi muncul.

// Step 1: createElement div, position: fixed, top: 0, z-index: 2147483647, height: 60px
// Step 2: requestFullscreen()
// Step 3: Saat fullscreen confirmed → div sudah menutupi notifikasi area
// Step 4: Ganti isi div dengan fake address bar
// Berbeda dari Teknik 1: blocker dipasang SEBELUM fullscreen (bukan di event handler)

Teknik 6 — Keyboard Intercept (ESC/F11)

Capture keydown event untuk ESC dan F11 sebelum browser memproses. ESC di level browser tidak bisa dicegah, tapi bisa dipakai untuk manipulasi UI state. F11 (Windows) bisa dicegah di beberapa browser.

// ESC: browser selalu keluar dari fullscreen — tidak bisa di-preventDefault
// F11: BISA di-preventDefault di Firefox (keyboard shortcut)
// Tapi: saat user pencet ESC, kita bisa detect → langsung re-enter fullscreen!
// Atau: saat ESC ditekan, hanya sembunyi fake UI, tapi tetap fullscreen
// User bingung: "koq ESC tidak works?" → retry → kita re-enter fullscreen

document.addEventListener('keydown', e => {
  if (e.key === 'Escape') {
    e.stopPropagation(); // tidak mencegah fullscreen exit, tapi mencegah handler lain
    // Langsung re-request fullscreen dari keyboard event (user gesture!)
    setTimeout(() => document.documentElement.requestFullscreen(), 50);
  }
  if (e.key === 'F11') {
    e.preventDefault(); // F11 BISA dicegah di beberapa browser
    e.stopPropagation();
  }
}, true); // useCapture: true — handler kita jalan sebelum lainnya
● Auto-aktif saat page load

Teknik 7 — Form-First Attack (Most Realistic 2025)

Tampilkan fake Google login form dulu (tanpa fullscreen). User mulai mengisi email → klik "Next" (user gesture!) → saat itu enter fullscreen. Notifikasi muncul tapi user sedang fokus mengisi password. Window distraction: 913–1366ms — cukup untuk form terlihat lebih "nyata" dari notifikasi.

// Phase 1: Show fake login form (normal window) — user percaya itu tab Google biasa
// Phase 2: User klik email field / klik "Next" = USER GESTURE
// Phase 3: requestFullscreen() dari click handler → selalu berhasil
// Phase 4: Notifikasi muncul TAPI user sedang fokus mengisi password
//          Chrome: 985ms window → user tidak sempat react
//          Opera:  913ms window → window paling sempit
//          Firefox: 1366ms window → paling lama
// Phase 5: Notifikasi auto-hilang, fake browser UI penuh terpasang
// Phase 6: User mengira mereka di tab Google asli
//
// Attack chain: credential phishing di dalam fullscreen spoof
// Tidak perlu cover notifikasi — cukup distract user selama <1.5 detik
